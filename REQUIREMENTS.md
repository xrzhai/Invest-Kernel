# Investment Agent v2 — 需求与设计原则

## 一、产品定位

**一句话**：面向长期价值投资者的 AI 投研与组合管理助手。

**关键词**：投资研究 / 组合管理 / 长期主义 / 价值投资。**不是**交易系统、量化策略引擎或 K 线技术分析工具。

**终极愿景**：可产品化的多用户平台。MVP 阶段先服务单用户（你自己），但架构上为多用户留好空间。

---

## 二、核心需求

### 2.1 Agent-First 架构

- Agent 是唯一入口，所有功能以 Tool 形式存在
- 用户通过自然语言与 Agent 交互，Agent 自主决定调用哪些 Tool、按什么顺序
- 前端无关：同一套 Agent 内核可接入 Web UI、飞书 Bot、CLI 等任意前端
- 可能演进为多 Agent 架构（具体形态在架构阶段讨论）

### 2.2 记忆系统

Agent 需要跨会话记忆，覆盖以下维度：

| 维度 | 说明 |
|------|------|
| 用户风格 | 投资偏好、沟通习惯（现 `profile.json` 的进化版） |
| 决策历史 | 过去的买卖决策、理由、结果 |
| 对话上下文 | 跨会话连续性（「上次我们聊到...」） |
| 市场洞察 | 累积的观察和经验 |
| 错误记忆 | 复盘教训（现 `mistake_memories` 的进化版） |
| Thesis 演变 | 不只是当前版本，而是为什么变、从哪变来的 |

参考方向：Hermes-Agent 等现代记忆系统设计。

### 2.3 投资研究

- **覆盖发起**：Agent 能自主完成对一家公司的初始研究，产出结构化 thesis
- **覆盖更新**：基于新信息（财报、新闻、事件）更新 thesis，记录变化原因
- **深度分析**（解决当前系统的四大短板）：
  - **丰富输入**：接入新闻、财报、研报、社交媒体等多种数据源
  - **灵活框架**：提供分析方向而非填空模板，给 LLM 更多思考自由度
  - **多轮迭代**：先思考 → 自我质疑 → 修正 → 再验证
  - **交叉验证**：主动搜索反面证据和反驳观点
- **可扩展**：能从 13 个标的扩展到 50+ 而不崩溃

### 2.4 组合管理

- 持仓管理：添加/删除/更新持仓
- 组合健康检查（日检）：刷新价格 → 规则检查 → 状态评估 → 行动建议
- 决策支持：用户说「我想买 XXX」，Agent 检查规则、评估风险、给建议
- 主动警报：IC 触发、大幅波动、规则违反时主动通知
- 自动化日检：可配置定时自动执行，用户只看报告

### 2.5 工作流与归档

- 所有决策、分析、日检自动归档，方便回溯
- 归档不只是存文件，而是结构化可查询的

### 2.6 可配置性

核心问题：现有系统的原则、模板、工作流都是硬编码的，无法泛化。

- 投资原则：平台提供框架 + 用户自定义内容
- 分析模板/框架：可配置，不同用户可以有不同的分析偏好
- 工作流：可定制（比如日检包含哪些步骤、什么频率）
- 规则引擎：可配置的 policy 规则（集中度上限、现金比例等）

### 2.7 数据源

- 可插拔的数据源架构（适配器模式）
- 初期继承 yfinance + JQData
- 未来扩展：新闻 API、财报解析、研报、社交媒体

### 2.8 LLM 策略

- 支持多模型（Claude / GPT / Gemini / 国产模型等）
- 任务路由：简单任务用快便宜的模型，深度分析用强模型
- 模型抽象层：业务逻辑与具体 LLM 解耦

## 三、设计原则

### P1: 确定性计算与 LLM 严格分离

所有数值计算（PnL、权重、CAGR、规则检查）必须由代码确定性完成。LLM 只负责解释、分析、建议等文字工作。这是从 v1 验证过的核心原则。

### P2: 去偏差设计

- `avg_cost`（成本价）不进入 LLM 决策 prompt，避免沉没成本偏差
- Invalidation Conditions 为首要约束，触发时无条件执行
- 主动搜索反面证据，防止确认偏差

### P3: Thesis-Driven

每个持仓必须有明确的投资论点（thesis）。没有 thesis 的持仓不应获得加仓建议。Thesis 是研究和决策的核心载体。

### P4: 单一真源（Single Source of Truth）

每一类数据只有一个权威来源。消灭现有系统中 Markdown 文件与 DB 之间的双写问题。其他视图都从真源派生。

### P5: 可插拔与可配置

- 数据源、LLM、前端都是可插拔的
- 投资原则、分析框架、工作流规则都是可配置的
- 为多用户/多策略做好扩展准备

### P6: Agent 自主性 + 人类监督

Agent 在预定义的能力范围内自主行动（选择 Tool、决定步骤），但关键决策（买卖执行）必须有人类确认环节。自主性和安全性的平衡。

### P7: 深度优于广度

分析质量优先于覆盖数量。宁可深入研究 10 家公司，也不要浅层覆盖 100 家。系统设计应鼓励深度思考（多轮迭代、交叉验证）而非快速填模板。

### P8: 可追溯性

所有决策、分析、建议都有归档和审计轨迹。不只是存下来，而是结构化、可查询、可复盘。支持「当时为什么做了这个决定」的回溯。

### P9: 渐进式能力增长

系统应该随着使用变得更好：积累记忆、学习用户风格、沉淀市场经验。不是每次从零开始的无状态工具。

## 四、待架构阶段讨论的开放问题

1. **多 Agent vs 单 Agent + 多 Tool**：具体的 Agent 拆分策略
2. **记忆系统实现**：存储选型、检索策略、遗忘机制
3. **数据架构**：DB 选型、结构化 vs 非结构化数据的组织方式
4. **部署模型**：本地 / 云 / 混合
5. **Coverage 体系重设计**：如何平衡结构化和灵活性
6. **技术栈选型**：语言、框架、依赖
7. **前端架构**：如何做到真正的前端无关
8. **实时性**：定时任务 vs 事件驱动 vs 混合

## 五、MVP 策略

**MVP 目标**：搭好 Agent + Tool 架构骨架，业务功能逐步迁入。

**MVP 包含**：
- Agent 核心框架（Tool 注册/调用/路由）
- LLM 抽象层（至少支持 2 个模型）
- 基础 Tool 集（组合查询、价格刷新、规则检查）
- 记忆系统原型
- 至少一种前端（CLI 或 Web）

**MVP 不含**：
- 全部数据源集成
- 多用户/租户
- 完整的覆盖研究流程
- 主动警报系统
